# NapFi — Confidential DeFi Savings Agent

NapFi lets a user sign in with Google or email (no wallet, no seed phrase), set a savings goal ("save 10 USDC every week"), and have an on-chain **ERC-8004 agent** automatically execute encrypted USDC deposits on a schedule. Balances are hidden using **Zama fhEVM** Fully Homomorphic Encryption. Scheduling is driven by Flow's **`FlowTransactionScheduler`**. Yield is generated by routing deposits through a **Uniswap v3 USDC/WETH** liquidity position. Every execution posts a verifiable **reputation receipt** on-chain.

**Tracks:** Flow (Consumer DeFi) · Zama (Confidential Finance) · ERC-8004 (Agents with Receipts)  
**Networks:** Ethereum Sepolia (Chain ID `11155111`) + Flow Testnet

## The Problem We Solve

### Problem 1 — DeFi balances are completely public

Every deposit you make is visible to anyone on a block explorer. Your balance, deposit amounts, and timing can be used to frontrun trades or identify and target you. **Zama fhEVM** solves this — all vault balances are stored as `euint64` (64-bit fully homomorphic encrypted integers). The contract runs `FHE.add()` and `FHE.sub()` on encrypted values without ever decrypting them. Only the user, through the Zama relayer SDK, can read their own balance.

### Problem 2 — No way to verify if an AI agent actually executed

If you delegate control of your money to an automated agent, there is no standard on-chain proof that it behaved correctly or has a trustworthy history. **ERC-8004** solves this — every NapFi agent is minted as an ERC-721 NFT on the `IdentityRegistry`, and every deposit execution posts a signed receipt (IPFS-pinned execution log) to the `ReputationRegistry`.

### Problem 3 — Automation still requires wallets, gas, and manual signing

Even simple DeFi automation requires the user to manage a wallet, hold ETH for gas, and manually sign transactions at execution time. **Flow's `FlowTransactionScheduler`** solves this — the server registers a scheduled Cadence transaction that fires automatically at the user's chosen interval. The user never signs anything after initial setup.

## Zama fhEVM — Confidential Balances

Zama's fhEVM lets Solidity contracts store and compute on encrypted integers using Fully Homomorphic Encryption (FHE). No one — not the contract owner, not a node operator, not even a full-node observer — can read user balances. NapFi uses this in two contracts.

### FHE Operations Used

| Operation | Where Used | What it does |
|-----------|------------|--------------|
| `FHE.fromExternal(handle, proof)` | deposit | Validates and converts the client-submitted ciphertext |
| `FHE.add(a, b)` | deposit | Adds two encrypted values without decrypting either |
| `FHE.sub(a, b)` | withdraw | Subtracts encrypted values |
| `FHE.le(a, b)` | withdraw | Encrypted comparison → returns `ebool` (encrypted boolean) |
| `FHE.select(cond, a, b)` | withdraw | Branch on encrypted boolean — safe zero-out on overdraft |
| `FHE.asEuint64(n)` | Uniswap vault | Convert plaintext to encrypted for share tracking |
| `FHE.isInitialized(val)` | both vaults | Check if a user has a balance entry |
| `FHE.allowThis(val)` | both vaults | Grant contract itself read permission (for future txs) |
| `FHE.allow(val, addr)` | both vaults | Grant a specific address read/decrypt permission |

### EncryptedVault.sol — Deposit

```solidity
// All user balances are euint64 ciphertext handles stored on-chain
mapping(address => euint64) private balances;

function deposit(externalEuint64 encryptedAmount, bytes calldata inputProof, address user) external {
    // 1. Validate ciphertext submitted by the Zama relayer SDK
    euint64 amount = FHE.fromExternal(encryptedAmount, inputProof);

    // 2. Homomorphic addition — both operands stay encrypted
    euint64 newBalance = FHE.isInitialized(balances[user])
        ? FHE.add(balances[user], amount)
        : amount;
    balances[user] = newBalance;

    // 3. Grant ACL: only user and agentOperator can decrypt
    FHE.allowThis(newBalance);
    FHE.allow(newBalance, user);
    FHE.allow(newBalance, agentOperator);
}
```

### EncryptedVault.sol — Withdraw (overdraft-safe without revealing balance)

```solidity
function withdraw(externalEuint64 encryptedAmount, bytes calldata inputProof) external {
    euint64 requested = FHE.fromExternal(encryptedAmount, inputProof);
    euint64 current   = balances[msg.sender];

    // Encrypted comparison — returns ebool, never reveals actual balance
    ebool hasEnough = FHE.le(requested, current);

    // FHE.select: if hasEnough → withdraw requested, else withdraw 0
    // Balance is never leaked even on a failed withdrawal attempt
    euint64 toWithdraw = FHE.select(hasEnough, requested, FHE.asEuint64(0));
    balances[msg.sender] = FHE.sub(current, toWithdraw);

    FHE.allowThis(balances[msg.sender]);
    FHE.allow(balances[msg.sender], msg.sender);
    FHE.allow(balances[msg.sender], agentOperator);
}
```

### Server-side Encryption (`server/src/lib/sepoliaExecutor.ts`)

The server uses `@zama-fhe/relayer-sdk` with `SepoliaConfig` — the same SDK as the browser — to produce real on-chain-verifiable ciphertexts:

```typescript
import { createInstance, SepoliaConfig } from "@zama-fhe/relayer-sdk/node"

// Convert JsonRpcProvider → EIP-1193 (required by relayer SDK)
const instance = await createInstance({ ...SepoliaConfig, network: jsonRpcToEip1193(provider) })

// Encrypt the USDC amount as euint64 (6 decimal micro-units)
const microUsdc = BigInt(Math.round(params.amountUSDC * 1_000_000))
const input = instance.createEncryptedInput(vaultAddr, operatorAddr)
input.add64(microUsdc)
const encrypted = await input.encrypt()

// Call vault — FHE.fromExternal verifies the proof on-chain
await vault.deposit(encrypted.handles[0], encrypted.inputProof, userAddr)
```

> Mock ciphertexts are rejected on-chain by `FHE.fromExternal`. Real Zama encryption is required for every deposit.

### Client-side Decryption (`frontend/lib/contracts.ts`)

```typescript
// 1. Get ciphertext handle from vault
const handle = await vault.getBalanceHandle(userAddress)  // bytes32

// 2. Decrypt via Zama relayer SDK — only the user's wallet can authorize this
const { decryptBalance } = await import("./zama")
const usdcAmount = await decryptBalance(handle, vaultAddr, userAddress, walletProvider)

// If ACL expired, re-grant then decrypt
await vault.makeBalanceDecryptable(user)  // FHE.allow(balance, user)
```

## Flow Actions + Flow Scheduler — Automation

NapFi uses Flow's `FlowTransactionScheduler` to create recurring Cadence transactions that execute automatically — no user action required after the initial setup.

### AgentScheduler.cdc (`cadence/contracts/AgentScheduler.cdc`)

Implements `FlowTransactionScheduler.TransactionHandler`. One `Handler` resource is stored per user in their Flow account. When `FlowTransactionScheduler` fires `executeTransaction()` at the scheduled time, the handler emits a `DepositTriggered` event:

```cadence
access(all) event DepositTriggered(
    userEVMAddress: String,   // Sepolia address to credit, e.g. "0xABC123..."
    amount:         UFix64,   // USDC to deposit, e.g. 10.0
    timestamp:      UFix64,   // block timestamp when fired
    executionId:    UInt64    // FlowTransactionScheduler job ID
)

access(FlowTransactionScheduler.Execute) fun executeTransaction(id: UInt64, data: AnyStruct?) {
    emit DepositTriggered(
        userEVMAddress: self.userEVMAddress,
        amount:         self.depositAmount,
        timestamp:      getCurrentBlock().timestamp,
        executionId:    id
    )
}
```

> **Flow Actions upgrade path:** The handler is designed to also run `FungibleTokenConnectors.VaultSource` (pull USDC from user's Flow wallet), optional `IncrementFiSwapConnectors.Swapper` (yield routing), and `FungibleTokenConnectors.VaultSink` (route to bridge relay). For the demo, the event emission drives the off-chain Sepolia execution via the Node.js server.

### `initHandler.cdc` — One-time Setup

Saves an `AgentScheduler.Handler` resource to `/storage/NapFiAgentSchedulerHandler` in the Flow account and publishes a public capability so `FlowTransactionScheduler` can call `executeTransaction()`. Accepts `userEVMAddress: String` and `depositAmount: UFix64`. Only runs once per agent.

### `scheduleDeposit.cdc` — Create Recurring Job

Creates a scheduled job on `FlowTransactionScheduler` (`0x8c5303eaa26202d6`) to fire at `getCurrentBlock().timestamp + delaySeconds`. Withdraws the scheduling fee from the account's FLOW balance.

| Frequency | Delay |
|-----------|-------|
| `daily` | 86,400 s |
| `weekly` | 604,800 s |
| `monthly` | 2,592,000 s |
| Demo | 60 s (via `FLOW_SCHEDULE_DELAY_SECONDS=60`) |

### Server FCL Integration (`server/src/lib/flowScheduler.ts`)

The server uses `@onflow/fcl` with **ECDSA_P256 + SHA3_256** signing (matching Flow faucet-created accounts):

```typescript
// SHA3-256 hash of raw tx bytes, then ECDSA_P256 sign with elliptic
function hashMsg(msgHex: string): Buffer {
  const sha = new SHA3(256)
  sha.update(Buffer.from(msgHex, "hex"))
  return sha.digest()
}

// Step 1: send initHandler.cdc — wait for seal
const initTxId = await fcl.mutate({
  cadence: loadCadenceTx("initHandler.cdc"),
  args: (arg, t) => [
    arg(params.userEVMAddress, t.String),
    arg(params.depositAmountUSDC.toFixed(8), t.UFix64),
  ],
  proposer: authz, payer: authz, authorizations: [authz], limit: 999
})
await fcl.tx(initTxId).onceSealed()

// Step 2: send scheduleDeposit.cdc — wait for seal
const scheduleTxId = await fcl.mutate({
  cadence: loadCadenceTx("scheduleDeposit.cdc"),
  args: (arg, t) => [
    arg(delaySecs.toFixed(8), t.UFix64),
    arg(String(1000), t.UInt64),   // execution effort limit
  ],
  proposer: authz, payer: authz, authorizations: [authz], limit: 999
})
await fcl.tx(scheduleTxId).onceSealed()
```

After each deposit completes, the server calls `scheduleNextFlowDeposit()` which sends a new `scheduleDeposit.cdc` for the next interval (skipping `initHandler` since the resource already exists).

### Flow Event Listener (`server/src/lib/flowListener.ts`)

The server HTTP-polls for `A.<schedulerAddr>.AgentScheduler.DepositTriggered` events. WebSocket transport is deliberately avoided because FCL's WS client does not attach an error listener — TLS resets (`ECONNRESET`) crash Node.

```typescript
// Poll in 249-block chunks (Flow Access API limit per request)
const MAX_BLOCK_SPAN = 249
// On first poll, scan back FLOW_EVENTS_LOOKBACK_BLOCKS (default 4000)
// to catch events missed during server downtime
while (lastEndHeight < latest) {
  const from = lastEndHeight + 1
  const to = Math.min(latest, from + MAX_BLOCK_SPAN)
  const decoded = await getEventsAtBlockHeightRange(eventType, from, to)
  // parse → enqueueFlowPendingDeposit()
  lastEndHeight = to
}
```

### Dashboard Auto-signing

When a `DepositTriggered` event arrives (or the time-based scheduler fires), the server enqueues a pending deposit. The `/dashboard` page polls `/api/flow-deposit-pending/:userAddress` every few seconds. When a pending deposit appears:

1. `depositUsdcFromWallet()` — embedded Web3Auth wallet silently signs `USDC.approve()` + `NapFiUniswapVault.depositUSDC()`. No user-facing popup.
2. `POST /api/flow-deposit-complete` — sends Sepolia tx hash back to server.
3. Server posts reputation receipt and reschedules next Flow execution.

## Uniswap v3 — Yield Generation

`NapFiUniswapVault.sol` is a single-sided USDC deposit vault. Users deposit only USDC; the vault handles all WETH internally. It provides full-range liquidity on the USDC/WETH 0.3% pool on Uniswap v3 Sepolia, earns LP fees, and tracks user positions with Zama FHE encryption.

### Two-Layer Accounting

| Layer | Storage | Why needed |
|-------|---------|------------|
| Plaintext | `totalShares`, `sharesOf[user]` | Uniswap periphery contracts require cleartext amounts for LP math |
| Encrypted | `encryptedBalances[user] euint64` | Private balance tracking — same FHE pattern as EncryptedVault |

### Deposit Flow

```solidity
function depositUSDC(uint256 amount) external {
    // 1. Pull USDC from user
    usdc.transferFrom(msg.sender, address(this), amount);

    // 2. Swap exactly half to WETH via SwapRouter02.exactInputSingle()
    //    User only ever touches USDC
    uint256 half = amount / 2;
    uint256 wethOut = swapRouter.exactInputSingle(ExactInputSingleParams({
        tokenIn:           address(usdc),
        tokenOut:          address(weth),
        fee:               3000,        // 0.3% USDC/WETH pool
        recipient:         address(this),
        amountIn:          half,
        amountOutMinimum:  0,
        sqrtPriceLimitX96: 0
    }));

    // 3. Increase the vault's Uniswap v3 position (full-range LP)
    //    tick range: -887220 to +887220 (tickSpacing=60 for 0.3% pools)
    npm.increaseLiquidity(IncreaseLiquidityParams({
        tokenId:        positionTokenId,   // vault holds the ERC-721 NFT
        amount0Desired: usdcForLP,
        amount1Desired: wethOut,
        amount0Min:     0, amount1Min: 0,
        deadline:       block.timestamp
    }));

    // 4. Update plaintext shares for proportional LP math
    totalShares += amount;
    sharesOf[msg.sender] += amount;

    // 5. Update encrypted balance — FHE.add, identical to EncryptedVault
    euint64 newBal = FHE.isInitialized(encryptedBalances[msg.sender])
        ? FHE.add(encryptedBalances[msg.sender], FHE.asEuint64(uint64(amount)))
        : FHE.asEuint64(uint64(amount));
    encryptedBalances[msg.sender] = newBal;
    FHE.allowThis(newBal);
    FHE.allow(newBal, msg.sender);
    FHE.allow(newBal, agentOperator);
}
```

### Withdrawal Flow

```solidity
function withdrawUSDC(uint256 shares) external {
    // 1. Proportional liquidity removal
    (, , , , , , , uint128 totalLiq, , , , ) = npm.positions(positionTokenId);
    uint128 liqToRemove = uint128((uint256(totalLiq) * shares) / totalShares);
    npm.decreaseLiquidity(DecreaseLiquidityParams({ tokenId, liquidity: liqToRemove, ... }));

    // 2. Collect removed tokens + any accrued LP fees
    (uint256 c0, uint256 c1) = npm.collect(CollectParams({
        tokenId, recipient: address(this),
        amount0Max: type(uint128).max, amount1Max: type(uint128).max
    }));

    // 3. Update shares
    sharesOf[msg.sender] -= shares;
    totalShares -= shares;

    // 4. Update encrypted balance — FHE.sub
    euint64 newBal = FHE.sub(encryptedBalances[msg.sender], FHE.asEuint64(uint64(shares)));
    encryptedBalances[msg.sender] = newBal;
    FHE.allowThis(newBal); FHE.allow(newBal, msg.sender); FHE.allow(newBal, agentOperator);

    // 5. Swap all received WETH back to USDC
    userUsdc += swapRouter.exactInputSingle(ExactInputSingleParams({
        tokenIn: address(weth), tokenOut: address(usdc), fee: 3000, ...
    }));

    // 6. Return USDC to user
    usdc.transfer(msg.sender, userUsdc);
}
```

## ERC-8004 — Agent Identity & Reputation

[EIP-8004](https://eips.ethereum.org/EIPS/eip-8004) defines trustless on-chain agents with verifiable identity and reputation. NapFi runs the full registration pipeline on first setup.

### Registration Pipeline (`server/src/lib/onchainSetup.ts`)

**Step 1 — Build EIP-8004 registration JSON:**
```typescript
{
  type: "https://eips.ethereum.org/EIPS/eip-8004#registration-v1",
  name: "NapFi-PrivyAgent",
  description: "Confidential DeFi savings agent — encrypted vault deposits on schedule",
  active: true,
  supportedTrust: ["reputation"],
  services: [{ name: "web", endpoint: "https://napfi.app" }],
  napfi: {
    userAddress,       // user's Sepolia wallet address
    goalAmountUSDC,    // e.g. 10
    frequency,         // "daily" | "weekly" | "monthly"
    yieldEnabled,      // true → NapFiUniswapVault, false → EncryptedVault
  }
}
```

**Step 2 — Pin to IPFS** via Lighthouse or Pinata. Returns `ipfs://CID`.

**Step 3 — `IdentityRegistry.register(ipfsUri)`:**
```typescript
const tx = await identity.getFunction("register(string)")(ipfsUri)
const receipt = await tx.wait()
// Parse Registered event logs to extract ERC-721 token ID
const agentId = ev.args.agentId  // e.g. BigInt(42)
```

**Step 4 — `setAgentWallet` with EIP-712 signature:**
```typescript
const types = {
  AgentWalletSet: [
    { name: "agentId",   type: "uint256" },
    { name: "newWallet", type: "address" },
    { name: "owner",     type: "address" },
    { name: "deadline",  type: "uint256" },
  ]
}
const signature = await operator.signTypedData(domain, types, {
  agentId, newWallet: operator.address, owner: ownerAddr, deadline
})
await identity.getFunction("setAgentWallet(uint256,address,uint256,bytes)")(
  agentId, operator.address, deadline, signature
)
// Links the operator wallet to the agentId — operator == EncryptedVault.agentOperator
```

**Step 5 — `AgentRegistry.registerUserAgent(user, agentId, vault)`:**  
Links the user's EOA to their `agentId` and vault. Enables `AgentRegistry.lookup(userAddress)` to hydrate agent state from chain without any off-chain database.

### Reputation Receipts (`server/src/lib/reputationPoster.ts`)

After every successful vault deposit, the server posts a receipt on-chain:

```typescript
// 1. Build execution log
const log = {
  timestamp, userAddress, depositAmountUSDC, sepoliaTxHash, flowTimestamp, status: "success"
}

// 2. Pin log to IPFS
const ipfsUri = await pinJsonToIpfs(log, `napfi-receipt-${Date.now()}.json`)

// 3. Content hash for on-chain integrity check
const feedbackHash = keccak256(toUtf8Bytes(JSON.stringify(log)))

// 4. Post to ERC-8004 ReputationRegistry
await registry.giveFeedback(
  BigInt(agentId),       // ERC-721 token ID
  BigInt(100),           // positive value score
  BigInt(0),             // value decimals
  "deposit_executed",    // tag1
  "success",             // tag2
  cfg.encryptedVault,    // endpoint (vault address as context)
  `ipfs://${cid}`,       // feedbackURI — IPFS execution log
  feedbackHash           // bytes32 content hash
)
```

## End-to-End User Flow

### 1. Social Login
User clicks "Get Started" → Web3Auth modal → Google/email login. Web3Auth uses threshold MPC key generation. The derived wallet works on both Sepolia (Chain ID `11155111`) and Flow Testnet. User never sees a seed phrase or private key.

### 2. Onboarding (`/onboarding`)
- **Step 1:** Enter USDC goal amount (e.g. `10`)
- **Step 2:** Select frequency (`Daily` / `Weekly` / `Monthly`)
- **Step 3:** Toggle yield optimization → click "Create My Agent"
- Calls `POST /api/setup`

### 3. Server: Agent Setup
1. Build + pin ERC-8004 registration JSON to IPFS
2. `IdentityRegistry.register(ipfsUri)` → mint agent NFT, capture `agentId`
3. EIP-712 sign + submit `setAgentWallet`
4. `AgentRegistry.registerUserAgent(user, agentId, vault)`
5. FCL: send `initHandler.cdc` (seal) then `scheduleDeposit.cdc` (seal)
6. Return `{ agentId, vaultAddress, nextExecutionISO, ipfsUri, txHashes, flowSchedule }`

### 4. Scheduled Execution (Flow)
At the scheduled time, `FlowTransactionScheduler` automatically calls `AgentScheduler.executeTransaction()`, which emits:
```
DepositTriggered(userEVMAddress: "0xABC...", amount: 10.0, timestamp: 1743500000.0, executionId: 7)
```

### 5. Server: Event Detection + Deposit
- `flowListener.ts` detects event via HTTP polling → enqueues pending deposit
- `/dashboard` polls `/api/flow-deposit-pending/:userAddress`
- When pending: `depositUsdcFromWallet()` silently signs `USDC.approve()` + `depositUSDC()` with Web3Auth embedded wallet (no popup)
- `POST /api/flow-deposit-complete` confirms with Sepolia tx hash
- Server: posts `ReputationRegistry.giveFeedback()` + reschedules next Flow execution

### 6. User Views Balance
- `vault.getBalanceHandle(user)` → `bytes32` ciphertext handle
- Zama relayer SDK: `decryptBalance(handle, vaultAddr, user, walletProvider)` → plaintext USDC amount
- Only the user's wallet can authorize decryption — the app and server cannot read it

## Smart Contracts

### EncryptedVault.sol

| Function | Description |
|----------|-------------|
| `deposit(externalEuint64, proof, user)` | `FHE.fromExternal` → `FHE.add` → update ACL |
| `withdraw(externalEuint64, proof)` | `FHE.le` encrypted comparison → `FHE.select` safe subtract |
| `getBalanceHandle(user)` | Returns `bytes32` ciphertext handle for Zama relayer decryption |
| `makeBalanceDecryptable(user)` | Re-runs `FHE.allow(balance, user)` for session refresh |
| `hasBalance(user)` | `FHE.isInitialized(balances[user])` |
| `updateAgentOperator(newOperator)` | Operator-only key rotation |

### NapFiUniswapVault.sol

| Function | Description |
|----------|-------------|
| `initializePosition(usdcAmount, wethAmount)` | Owner seeds initial LP; sets `positionTokenId` |
| `depositUSDC(amount)` | Swap half→WETH → `npm.increaseLiquidity()` → update plaintext shares + `FHE.add` |
| `withdrawUSDC(shares)` | `npm.decreaseLiquidity()` → `npm.collect()` → swap WETH→USDC → transfer USDC → `FHE.sub` |
| `getUsdcBalance(user)` | Plaintext `sharesOf[user]` (USDC-denominated, excludes unrealized fees) |
| `getBalanceHandle(user)` | Encrypted balance handle — same API as EncryptedVault |
| `getPositionLiquidity()` | Current LP liquidity from `npm.positions(positionTokenId)` |

### AgentRegistry.sol

| Function | Description |
|----------|-------------|
| `registerUserAgent(user, agentId, vault)` | Links EOA → agentId + vault; emits `AgentRegistered` |
| `lookup(user)` | Returns `(agentId, vaultAddress, isRegistered, registeredAt)` |
| `getUserByAgent(agentId)` | Reverse lookup |
| `batchLookup(users[])` | Fetch multiple users at once |

## HTTP API Reference

Base URL: `http://localhost:3001`. Set `NEXT_PUBLIC_API_BASE_URL` in `frontend/.env.local`.

### Health

| Method | Path | Response |
|--------|------|----------|
| `GET` | `/` | Service info, endpoint list, `onchainReady`, `depositQueueSource` |
| `GET` | `/health` | `{ ok, onchainReady, depositQueueSource }` |

### Agent Lifecycle

| Method | Path | Body / Params | Response |
|--------|------|---------------|----------|
| `POST` | `/api/setup` | `{ userAddress, goalAmountUSDC, frequency, yieldEnabled }` | `{ agentId, vaultAddress, nextExecutionISO, ipfsUri, txHashes, flowSchedule? }` |
| `GET` | `/api/agent/:userAddress` | — | Full agent state or **404** |

**`POST /api/setup` logic:**
- **New user:** full IPFS → ERC-8004 → AgentRegistry → Flow schedule pipeline
- **In-memory already:** updates `goalAmountUSDC` only
- **On-chain but not in RAM:** hydrates from `AgentRegistry.lookup()`, stores in RAM, returns
- **503** if first-time setup requires on-chain env that's missing

### Receipts

| Method | Path | Response |
|--------|------|----------|
| `GET` | `/api/receipts/:agentId` | `{ receipts: [{ timestamp, amountUSDC, sepoliaTxHash, flowTimestamp }] }` |

### Flow ↔ Dashboard Deposit Queue

| Method | Path | Body | Response |
|--------|------|------|----------|
| `GET` | `/api/flow-deposit-pending/:userAddress` | — | `{ pending: null \| { id, amountUSDC, flowTimestamp, vaultAddress } }` |
| `POST` | `/api/flow-deposit-complete` | `{ id, userAddress, sepoliaTxHash }` | `{ ok, receipt }` |

### Demo Endpoints (default on, `NAPFI_DEMO_ENDPOINTS=true`)

| Method | Path | Body | Description |
|--------|------|------|-------------|
| `POST` | `/api/demo/schedule-one-minute` | `{ userAddress }` | Sends `scheduleDeposit.cdc` with 60s delay. **503** if Flow not configured. |
| `POST` | `/api/demo/execute-deposit` | `{ userAddress }` | Encrypt + deposit + reputation, no Flow trigger required. |

### Cron

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `POST` | `/api/cron/enqueue-schedule-deposits` | `Bearer <NAPFI_CRON_SECRET>` | Trigger time-based deposit enqueue. Only when `NAPFI_DEPOSIT_QUEUE_SOURCE=time`. |

### Deposit Queue Modes

| `NAPFI_DEPOSIT_QUEUE_SOURCE` | Behavior |
|------------------------------|----------|
| `time` (default) | Server polls `nextExecutionISO`; enqueues when wall clock passes it |
| `flow` | Enqueues only when `DepositTriggered` event arrives from Flow testnet |

## Frontend Pages

### `/` — Landing Page
Hero, tech stack integrations, feature overview. No wallet connection button.

### `/onboarding` — Setup Wizard
Step 1: USDC goal amount → Step 2: Frequency → Step 3: Yield toggle + confirm. Calls `POST /api/setup`. Shows loading while agent is registered on-chain and Flow schedule is created. Redirects to `/dashboard`.

### `/dashboard` — Main App
- **Wallet USDC balance** — ERC-20 `balanceOf`
- **Encrypted vault balance** — `getBalanceHandle()` + Zama relayer SDK decryption
- **LP info panel** — user share %, total pool TVL, position token ID, liquidity, uncollected fees (USDC + WETH)
- **Deposit** — `approve()` + `depositUSDC()` via embedded wallet (silent, no popup)
- **Withdraw** — `withdrawUSDC(shares)` removes LP + swaps WETH → USDC + returns USDC
- **Countdown** — live timer to `nextExecutionISO`
- **Flow auto-sign** — polls pending queue every ~5s; silently executes when deposit is due
- **Demo buttons** — "Schedule 1-minute test" + "Execute now" (bypass Flow for demos)

### `/receipts` — Execution History
Paginated receipts: date, amount, Flow trigger timestamp, Sepolia tx hash (Etherscan link), total USDC saved.

### `/settings` — Account Management
Update goal (amount + frequency + yield toggle) → `POST /api/setup`. Vault address + Etherscan link. Withdraw modal. Wallet address copy. Logout.

## Environment Variables

### Server (`server/.env`)

| Variable | Required | Description |
|----------|----------|-------------|
| `SEPOLIA_RPC_URL` | Yes | Ethereum Sepolia RPC (Alchemy / Infura / public) |
| `BACKEND_PRIVATE_KEY` | Yes | Funded Sepolia wallet — sends all on-chain txs |
| `OPERATOR_PRIVATE_KEY` | No | Separate operator key; falls back to `BACKEND_PRIVATE_KEY` |
| `LIGHTHOUSE_API_KEY` | One of | IPFS pinning via Lighthouse |
| `PINATA_JWT` | One of | IPFS pinning via Pinata |
| `VAULT_CONTRACT_ADDRESS` | No | Override vault (uses `contracts.json` `encryptedVault` if unset) |
| `FLOW_ACCESS_NODE` | For Flow | e.g. `https://rest-testnet.onflow.org` |
| `FLOW_ACCOUNT_ADDRESS` | For Flow | Flow account that deployed AgentScheduler |
| `FLOW_PRIVATE_KEY` | For Flow | Hex private key (ECDSA_P256) |
| `FLOW_KEY_INDEX` | No | Key index (default `0`) |
| `AGENT_SCHEDULER_ADDRESS` | For Flow | `0xc4d59c93cd6c2c43` |
| `FLOW_SCHEDULE_DELAY_SECONDS` | No | Override schedule delay (e.g. `60` for demo) |
| `FLOW_EVENTS_POLL_MS` | No | Event poll interval in ms (default `10000`) |
| `FLOW_EVENTS_LOOKBACK_BLOCKS` | No | Blocks to scan on first poll (default `4000`) |
| `NAPFI_DEPOSIT_QUEUE_SOURCE` | No | `time` (default) or `flow` |
| `NAPFI_DEMO_ENDPOINTS` | No | `true` / `false` (default `true`) |
| `NAPFI_CRON_SECRET` | No | Bearer token for cron endpoint |
| `AGENT_ID` | For reputation | ERC-8004 token ID for `giveFeedback` |
| `CORS_ORIGIN` | No | Allowed frontend origin |
| `PORT` | No | Server port (default `3001`) |

## Running Locally

### 1. Server
```bash
cd server
npm install
cp .env.example .env   # fill SEPOLIA_RPC_URL, BACKEND_PRIVATE_KEY, IPFS keys
                       # add FLOW_* vars to enable Flow scheduling
npm run dev            # starts on port 3001
```

### 2. Frontend
```bash
cd frontend
pnpm install
echo "NEXT_PUBLIC_API_BASE_URL=http://localhost:3001" > .env.local
pnpm dev               # starts on port 3000
```

### 3. Onchain (redeploy if needed)
```bash
cd onchain
npm install
npx hardhat compile
npx hardhat run scripts/deploy.ts --network sepolia
# Update contracts.json with new addresses
```

### 4. Demo Mode (no Flow)
Flow env vars are optional. To test the full Zama encryption + deposit + reputation pipeline without Flow:
```bash
curl -X POST http://localhost:3001/api/demo/execute-deposit \
  -H "Content-Type: application/json" \
  -d '{"userAddress": "0xYourSepoliaAddress"}'
```
